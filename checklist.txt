HARNESS FIX (done)
================================================================================
Root cause of the transient c34001 "BIND:unresolved_symbols" skips: the five
`timeout 2 llvm-link ...` sites used a 2-second cap. Under 16-way parallelism
with a cold page cache (first run after `make`), llvm-link's first-touch disk
reads blow past 2s, `timeout` SIGKILLs it (exit 124/137), and the harness
mislabels that kill as "BIND:unresolved_symbols" -> a false *skip*. A genuine
unresolved-symbol error (a real compiler/runtime defect) and a load-induced
kill were indistinguishable, so real skips could not be trusted.

Fix (run_acats.sh):
- Factored the 5 duplicated link invocations into one `link_program` helper
  that sets LINK_STATUS = ok | timeout | unresolved from llvm-link's exit code.
- Raised the cap to LINK_TIMEOUT (default 20s, env-overridable).
- A timeout is now a distinct `fail ... TIMEOUT:llvm-link` line, never a skip.
  Class L (expected link rejection) no longer counts a timeout as a pass.
- Net: fewer lines, DRY, and the skip list is now trustworthy.

FOLLOW-ON TARGET: ce2 (Chapter 14 I/O) — the C-group with the most failures (9)
================================================================================
Three root causes, all fixed:

1. FCB slot exhaustion (ce2117a/b) [rts/sequential_io.adb, rts/direct_io.adb]:
   Next_FCB advanced on every create/open and was never reclaimed on
   close/delete, so 99 CUMULATIVE opens raised USE_ERROR even with few files
   open. Reuse the first closed slot; USE_ERROR only when all are open.

2. Cross-handle write visibility (ce2111h) [rts/direct_io.adb]: WRITE left the
   datum in the stdio buffer, so a second internal file on the same external
   file read stale data -> END_ERROR. Flush after each write.

3. File continuity (ce2108b/d/f/h, and ce3112b/d) [run_acats.sh]: a reader test
   opens a file left by a creator it names via LEGAL_FILE_NAME(_, "CExxx"). The
   ACATS model runs the creator first in the same directory; our per-test CWD
   isolation hid it. run_continuity_creators compiles+links+runs each named
   creator into the reader's own lib dir first.

4. DATA_ERROR vs CONSTRAINT_ERROR on invalid read (ce2205b, ce2405a) [ada83.c]:
   THE BIG ISSUE. Reading an out-of-range scalar (enum written by one
   instantiation, read by another) raised CONSTRAINT_ERROR on the generic
   READ's out-param copy-back. RM 12.3 shared-generic view: SEQUENTIAL_IO's
   body was compiled against the constraint-free PRIVATE view of ELEMENT_TYPE,
   so no scalar copy-back check exists; our macro instantiation wrongly
   re-derived it from the actual (LAKES/MEN). Fix: flag the instance actual-view
   of a formal PRIVATE type (Type_Info.formal_private_view) and skip the
   RM 6.4.1 scalar copy-back check for such parameters. The read then stores the
   raw value with no exception (RM 14.2.2 permits omitting the DATA_ERROR check).
