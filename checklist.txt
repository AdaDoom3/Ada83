ROOT-CAUSE NOTES (upstream analysis)  -- RM 4.2, RM 4.3.2, RM 3.6.3
═══════════════════════════════════════════════════════════════════════════════
RM 4.2: a string literal's bounds are those of a positional array aggregate
        (4.3.2). Type is determined solely from context.
RM 4.3.2 (positional, no others): the lower bound is
   * the APPLICABLE INDEX CONSTRAINT's lower bound, if the aggregate/literal
     appears in one of contexts (a),(b),(c);
   * OTHERWISE  S'FIRST, where S is the array type's index SUBTYPE.
   The upper bound follows from the component count.
Contexts (a)-(c) (which supply an applicable index constraint):
   (a) actual param / generic actual / function result / RHS of ":=", AND the
       corresponding subtype is a CONSTRAINED array subtype;
   (b) operand of a qualified expression whose type mark is a constrained
       array subtype;
   (c) the expression of a component association of an enclosing (array or
       record) aggregate (multidim subaggregates inherit the context).
Parentheses BREAK contexts (a)-(c): a parenthesized primary is a general
   expression, so the string literal inside gets the "otherwise" rule
   (lower = S'FIRST). Its value is then checked against the surrounding
   subtype -> may raise CONSTRAINT_ERROR (this is c42007f).
RM 3.6.3 (catenation): result low bound = low bound of the left operand when
   the left operand is non-null; hence a "&"-operand string literal is NOT in
   a context (a)-(c), so it takes S'FIRST (this is c42007e).

Why siblings a-d,g-k already pass: they exercise contexts (a)-(c) where we
already take bounds from the target constraint. The hole is precisely the
"otherwise -> S'FIRST" branch, where we hardcode lower bound = 1.

CONFIRMED MECHANISM (traced through the compiler):
  Generate_String_Literal (ada83.c ~25315) ALREADY takes the index subtype's
  'FIRST when node->type is the applicable array type. The bug is upstream in
  RESOLUTION: the string-literal operand of "&" is committed to STRING before
  the STR context is known, so it never reaches that codegen branch.
    - Analyze_Expr eagerly resolves a bare NK_STRING to STRING (13712), unlike
      NK_AGGREGATE which is deferred as a flexible operand (13493).
    - The catenation codegen (27089+) already computes result bounds correctly
      (RM 4.5.3: result low = left operand's own low bound, null-left -> right).
      So fixing the operand's type is sufficient; codegen needs no change.

  c42007e FIX (done-ish): seed flexible concat operands (NK_STRING / aggregate)
    with Type_Base(ctx) in Resolve_Binary_Op, so they carry the applicable
    array type (index subtype -> 'FIRST). Concat operands are never in an
    applicable-index-constraint context (a)-(c), so the UNCONSTRAINED base is
    always right -> S'FIRST.

  c42007f FIX (harder): the parser DISCARDS parentheses around a non-aggregate
    primary (Parse: line ~6153 returns the inner expr; only NK_AGGREGATE gets
    is_parenthesized). So ("WHO") becomes bare NK_STRING and we lose the fact
    that it must use S'FIRST (parentheses remove it from context (c)). Fix:
    preserve parenthesization for string literals, and when propagating a
    context array type into a parenthesized literal, propagate the
    UNCONSTRAINED base so it takes S'FIRST; the existing belongs-to-subtype
    check then raises CONSTRAINT_ERROR (value 2..4 vs component subtype 3..5).

UNIFYING RULE: propagate the unconstrained base (-> S'FIRST) to a string
  literal / positional aggregate UNLESS it sits DIRECTLY (no parentheses, no
  "&") in an applicable-index-constraint context (a)-(c).

Target failure: c67005b — "if equality is redefined for a scalar type, case
statements still use predefined equality". All three sub-failures were one root
cause: a comparison of a value of MY (TYPE MY IS NEW INTEGER) wrongly resolved
to a "=" redefined for INTEGER. MY is a DISTINCT type (RM 3.4) that keeps the
predefined equality inherited at the point of derivation; a "=" declared for
INTEGER after the derivation is not MY's.

Root cause: Type_Covers lets a type and a type derived from it cover each other
(so shared integer/enumeration literals resolve), and Type_Base collapses a
derived type onto its parent's representation. So the user INTEGER "=" matched
MY operands in overload resolution.

Fix: Named_Type walks the subtype (base_type) chain but stops at a derivation
boundary (parent_type set) — a subtype of INTEGER resolves to INTEGER's base,
while MY resolves to MY. Formal_Accepts_Actual matches a user operator's formal
to an actual by exact named type (universals still convert via Type_Covers).

Scope: applied to the EQUALITY operators ("=", "/=") only. parent_type is
overloaded — CALENDAR.TIME is represented THROUGH DURATION (TIME->parent_type =
DURATION), so treating them as distinct broke the CALENDAR "+"/"-" resolution
(c96005e) that relies on the loose coverage. Equality is exactly the RM 6.7
operator whose redefinition must not cross a derivation boundary, so scoping
there is correct and leaves arithmetic on the existing permissive path.

Gate: full Class C — c67005b FAIL->PASS, zero regressions (c97307a is the
known flaky tasking test).
