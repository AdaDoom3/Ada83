WORK LOG
--------
1. Survey run (test_results/c-20260717-152813-4406): c44 was the only
   group with 100% of its tests skipped. All five died the same way:
   llvm-link rejected the IR with `use of undefined value '%t0'`,
   preceded by `warning: Generate_Aggregate: unhandled aggregate type
   kind=1` (kind 1 = TYPE_BOOLEAN).

2. Root cause #1 — Analyze_Binary flexible-operand adoption (§12).
   In `(I1 MOD I2 <= I5) /= (1 .. 3 => 7)` the inner "<=" legitimately
   carries two interpretations: predefined "<=" -> BOOLEAN and the user
   "<="(ARR,ARR) -> ARR. The aggregate operand has no interpretations of
   its own (RM 4.3 flexible operand), and Analyze_Binary let it adopt the
   sibling's candidate types WHOLESALE — including BOOLEAN. That built a
   spurious "/="(BOOLEAN,BOOLEAN) interpretation which Select_Interp
   picked, driving the aggregate into a scalar type it cannot denote; the
   expander then had nothing to emit and produced the %t0 placeholder.
   FIX (upstream, in resolution, not the expander): adoption is now
   bounded by what the literal form can possibly denote (RM 8.6
   applicable interpretations): an aggregate only a composite type
   (RM 4.3), a string literal only a one-dimensional character array
   (RM 4.2). Nested predicate Flexible_Can_Denote in Analyze_Binary.
   Result: c44003a/d/f/g pass; c44003e progressed to a runtime FAIL.

3. Root cause #2 — Select_Interp universal preference (§12).
   c44003e section 3 (fixed-point USR with user "AND" ">=" "+" "-" "/"
   "**"): `-3.5 - 11.0` in a USR context computed -14.5 instead of -3.5.
   The candidates for the binary "-" are (universal_real) and the
   predefined "-"(USR,USR) whose operand type was contributed by the
   nested USER unary "-"(USR) interpretation. Step 3 of Select_Interp
   (RM 4.10 universal preference) returned the universal interpretation
   because its escape clause only recognized USER-NAMED candidates
   matching the context. But per RM 4.6 an implicit universal conversion
   applies iff there is no legal interpretation without it, and only a
   literal, named number, or attribute is a convertible operand — an
   operator RESULT is not. So a universal interpretation can never
   legally satisfy a concrete numeric context; the literals convert
   instead and the concrete operation governs (which then resolves the
   nested unary minus to the user "-"(USR) -> 7.5 -> 7.5-11.0 = -3.5).
   FIX: the escape now recognizes ANY concrete candidate denoting the
   context type — user-declared or predefined — while keeping the
   NEW-derivation exclusion (c87b07d) and the no-context / universal-
   context behavior (c87b07a) intact.

4. c44 group: 5/5 PASSED.
